1. Résumé exécutif

Ce rapport propose une découpage logique du développement d’une plateforme e-commerce de type marketplace (clients, vendeurs, administrateurs) selon la méthode Agile Scrum, sur une durée de six mois (janvier à juin 2025).

Les principes retenus :

  • Un Epic majeur par mois, assimilé à une release (livraison incrémentale utilisable).
  • Deux sprints par mois de deux semaines chacun → 12 sprints sur l’ensemble de la période.
  • Chaque sprint est décrit comme en entreprise : objectifs, backlog, cérémonies, livrables et critères de fin (Definition of Done).

Le périmètre fonctionnel s’appuie sur les modules réellement présents dans le dépôt (authentification, catalogue, panier, commandes, avis, suivi de vendeurs, administration, tableau de bord, chat, etc.).

2. Contexte et périmètre du projet

2.1 Vision produit

Offrir un marché en ligne où :

  • Les clients parcourent un catalogue, consultent les boutiques des vendeurs, gèrent un panier, passent commande, laissent des avis et suivent des vendeurs.
  • Les vendeurs gèrent leurs produits, catégories propres à la boutique, commandes, commentaires et audience (abonnés).
  • Les administrateurs supervisent utilisateurs, marchés, catégories globales, commandes/commentaires selon les écrans prévus et assurent la cohérence de la plateforme.

2.2 Acteurs (personas)

Acteur Besoins principaux
Client Découverte, confiance (avis), panier fluide, checkout, compte
Vendeur CRUD produits, visibilité, relation client (commentaires), suivi des ventes
Admin Gouvernance, modération, statistiques, paramétrage

2.3 Hors périmètre (exemples à cadrer en début de projet)

  • Moyens de paiement réels (intégration bancaire PCI-DSS) si non requis académiquement — souvent simulé en phase projet.
  • SLA production 24/7, sauvegarde externalisée — à traiter en fin de cycle ou hors scope.

3. Architecture technique (synthèse)

Couche Technologie Rôle
Front-end Vue.js 3, Vue Router, Bootstrap UI, navigation par rôles (layouts Auth / Default / Admin / Seller)
Back-end NestJS, TypeORM API REST, modules métier
Base de données MySQL Persistance entités (utilisateur, catégorie, produit, panier, commande, avis, etc.)
Transversal JWT / auth, upload, mail, i18n Sécurité, médias, notifications, internationalisation

Cette architecture justifie un découpage en Epics alignés sur les modules backend et les parcours utilisateur front.

4. Méthodologie Agile Scrum

4.1 Rôles Scrum (adaptation projet étudiant / petite équipe)

Rôle Responsabilité
Product Owner (PO) Priorise le backlog, valide les releases, rédige les user stories
Scrum Master Facilite les rituels, enlève les blocages, protège l’équipe
Équipe de développement Implémentation, tests, documentation technique

En contexte ING2, un même étudiant peut cumuler développeur et PO sur un mini-projet ; en équipe, la répartition est recommandée.

4.2 Artefacts Scrum

  • Product Backlog : liste ordonnée de fonctionnalités et corrections.
  • Sprint Backlog : sous-ensemble du backlog engagé pour le sprint courant.
  • Incrément : version potentiellement livrable à la fin de chaque sprint.

4.3 Cérémonies (rythme bi-hebdomadaire)

Cérémonie Durée indicative Objectif
Sprint Planning 2–4 h Choisir les items, découper en tâches, estimer
Daily Scrum 15 min / jour Synchronisation : hier / aujourd’hui / blocages
Sprint Review 1–2 h Démo de l’incrément aux parties prenantes
Rétrospective 1 h Amélioration continue du processus

4.4 Definition of Done (exemple projet)

Une user story est « terminée » lorsque :

  1. Le code est revu (pair review ou auto-checklist).
  2. Les tests manuels ou automatisés couvrent le cas nominal et un cas d’erreur.
  3. L’API est documentée (Swagger ou README module) si nouvelle route.
  4. La build front et back passe sans erreur.
  5. La fonctionnalité est démontrable en environnement de staging / local unifié.

6. Cartographie Epics ↔ Releases

Chaque Epic correspond à une release mensuelle (objectif métier majeur livré ou consolidé à la fin du mois).

Mois Epic / Release Thème Modules / domaines principaux

Février R2 – Catalogue & offre vendeur Catégories, produits, médias Category, Product, Image, Upload, vues Catalog, Shop, CRUD vendeur
Mars R3 – Panier & commandes Parcours d’achat Cart, Order, vues Cart, Checkout

7. Détail des sprints (mode entreprise)

Pour chaque mois, Sprint A = semaines 1–2, Sprint B = semaines 3–4.
Les user stories sont formulées au format : En tant que …, je veux … afin de ….

Epic R1 – Janvier 2025 · Fondations & identité

Sprint 1 (semaines 1–2)

Élément Contenu
Objectif sprint Monorepo / deux dépôts opérationnels, CI basique, première API « health », modèle utilisateur et auth minimale
User stories (exemples) US1.1 : En tant qu’équipe technique, je veux un dépôt back configuré avec TypeORM et MySQL afin de persister les données. · US1.2 : En tant que client, je veux créer un compte avec rôle afin d’accéder aux fonctionnalités adaptées. · US1.3 : En tant qu’utilisateur, je veux me connecter avec JWT afin d’accéder aux routes protégées.
Tâches techniques Schéma User ; endpoints register/login ; middleware JWT ; .env.example ; CORS
Livrables API auth fonctionnelle ; écran Sign-in / Sign-up branché ; document d’installation
Démo review Création compte + login + accès route protégée (Postman + UI)

Sprint 2 (semaines 3–4)

Élément Contenu
Objectif sprint Finaliser rôles (client / seller / admin), redirections Vue Router, mot de passe oublié si scope
User stories US1.4 : En tant qu’admin, je veux être redirigé vers le dashboard admin après login. · US1.5 : En tant que vendeur, je veux accéder à l’espace vendeur sans voir l’admin. · US1.6 : En tant qu’utilisateur, je veux réinitialiser mon mot de passe (flux mail ou code).

8. Gestion des risques et qualité

Risque Impact Mitigation
Retard sur l’auth Bloque tout le reste Prioriser Epic R1 ; feature freeze sur le scope secondaire
Complexité upload / Cloudinary Médias instables Spike technique en début de Sprint 4
Scope commandes / paiement Attentes irréalistes Paiement « mock » documenté pour le jury
Dette technique Ralentissement fin de période Rétrospectives honnêtes ; refactors localisés

Stratégie de test : tests unitaires sur services NestJS critiques ; tests manuels par user journey ; checklist de non-régression avant chaque review de sprint.

9. Indicateurs (KPI) et clôture

Indicateurs simples adaptés à un projet académique :

  • Vélocité : nombre de story points réalisés par sprint (après 3 sprints, ajuster le planning).
  • Taux de Done : % des stories du sprint réellement « Done » en fin de sprint.
  • Bugs ouverts : nombre de défauts critiques non résolus (objectif : 0 en fin de release).
  • Couverture des parcours : % des parcours persona couverts par une démo.

Clôture juin 2025 : livraison du backlog prioritaire, rapport de bilan, démo finale, transfert de connaissance (README, variables d’environnement).

Annexe A – Glossaire rapide

  • Epic : grande fonctionnalité métier, découpée en stories.
  • Release : version livrée aux utilisateurs ou au jury, avec numéro de version.
  • Sprint : itération courte avec objectif clair et incrément livrable.
  • INVEST : critères pour de bonnes user stories (Indépendante, Négociable, Valuable, Estimable, Small, Testable).
